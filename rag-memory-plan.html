RAG Memory Plan v2 WhatsApp + n8n + Qdrant 繁中 / 粵語友好

WhatsApp 群組 Shared RAG Memory 改良計劃

這份版本保留原本方案的核心方向，並補強訊息 ID、錯誤處理、Qdrant 初始化、Docker 啟動順序、成本估算、保留策略與實作驗收標準。

1. 核心目標

建立一個「群組共同記憶 AI」。指定 WhatsApp 群組內的有效文字訊息會被記錄到向量資料庫；當任何人用 @ai 發問時，AI 會查詢同一個群組的共同記憶，再根據相關歷史內容回答。

A: 我星期五要去深圳
B: @ai A 星期五去哪？
AI: 根據群組之前的聊天，A 星期五要去深圳。
重點：這是 group-level shared memory，不是 per-user memory。B 可以查到 A 曾在同一群組說過的內容。

2. 整體架構

WhatsApp Group → whatsapp-bridge → n8n → OpenAI Embeddings → Qdrant → DeepSeek → WhatsApp 回覆

n8n

負責 workflow、分支判斷、呼叫 Embedding API、寫入 Qdrant、查詢 Qdrant、組裝 RAG prompt、呼叫 DeepSeek。

whatsapp-bridge

負責接收 WhatsApp 訊息，判斷群組與指令，將普通訊息或 @ai 指令送到 n8n。

Qdrant

本機 Docker 內的向量資料庫，用來做語意搜尋。n8n 透過 http://qdrant:6333 呼叫。

OpenAI + DeepSeek

OpenAI text-embedding-3-small 負責文字轉向量；DeepSeek 負責根據 RAG context 產生回答。

4. whatsapp-bridge 改良邏輯

目前 bridge 只 forward @ai / @aiimg。RAG 版本需要改成普通群訊息也 forward，但普通訊息只記錄，不回覆。

訊息類型 command n8n 行為
普通群文字 record 清理文字 → embedding → Qdrant upsert，不回覆
@ai 問題 chat 問題 embedding → Qdrant search → DeepSeek → 回覆 WhatsApp
@ai memory memory 可選：總結近期群組記憶狀態
@aiimg prompt image 走原本生圖 / 改圖流程
重要：一定要避免記錄 bot 自己送出的 AI 回覆，否則記憶會被 AI 自己的答案污染，之後回答可能越來越偏。

5. 記憶資料設計

每條有效訊息會變成 Qdrant 的一個 point。建議 payload 加入 messageId、source、expiresAt，方便去重、追蹤與清理。

{
  "id": "whatsapp-native-message-id-or-hash",
  "vector": [0.01, 0.22, 0.33],
  "payload": {
    "messageId": "whatsapp-native-message-id-or-hash",
    "groupId": "852xxx@g.us",
    "groupName": "Private Wutsapp Group",
    "userId": "111702765606:38@lid",
    "userName": "Kelvin",
    "text": "我星期五要去深圳",
    "timestamp": 1780222830,
    "expiresAt": 1795774830,
    "type": "whatsapp_message",
    "source": "whatsapp-bridge"
  }
}
欄位 用途
messageId用 WhatsApp 原始 message ID；如果拿不到，使用 groupId + userId + timestamp + text 產生 hash。
groupId限制只搜尋同一個 WhatsApp 群組，避免不同群組資料混在一起。
userId / userName知道訊息是誰說的，回答時可以引用人名。
timestamp顯示日期、排序、做近期優先。
expiresAt支援 180 天保留策略，未來可做定期清理。

6. Embedding 選擇與成本

第一版建議使用 OpenAI text-embedding-3-small，向量大小 1536，Qdrant collection 必須使用相同 size。

PUT http://qdrant:6333/collections/whatsapp_memory

{
  "vectors": {
    "size": 1536,
    "distance": "Cosine"
  }
}

成本估算

text-embedding-3-small 約 US$0.02 / 1M tokens。假設每日 1000 條訊息、每條平均 50 tokens，每月約 1.5M tokens，embedding 成本大約 US$0.03 / 月。

第一版策略

先每條有效文字訊息都 embedding。小群組成本低，做法簡單，最容易驗證 RAG 流程。

第二版優化

如訊息量增加，可把同一人連續短訊息合併成 chunk，或每 30-60 秒批量處理一次。

7. Qdrant API 流程

7.1 建立 collection

PUT http://qdrant:6333/collections/whatsapp_memory

{
  "vectors": {
    "size": 1536,
    "distance": "Cosine"
  }
}

7.2 寫入記憶

PUT http://qdrant:6333/collections/whatsapp_memory/points?wait=true

{
  "points": [
    {
      "id": "message-id",
      "vector": [0.01, 0.22, 0.33],
      "payload": {
        "groupId": "852xxx@g.us",
        "userId": "111702765606:38@lid",
        "userName": "Kelvin",
        "text": "我星期五要去深圳",
        "timestamp": 1780222830,
        "expiresAt": 1795774830,
        "type": "whatsapp_message"
      }
    }
  ]
}

7.3 搜尋同群記憶

POST http://qdrant:6333/collections/whatsapp_memory/points/search

{
  "vector": [0.01, 0.22, 0.33],
  "limit": 8,
  "filter": {
    "must": [
      {
        "key": "groupId",
        "match": {
          "value": "852xxx@g.us"
        }
      }
    ]
  },
  "with_payload": true
}

8. n8n Workflow 設計

record 分支

  1. Webhook 收到 command: record
  2. 檢查 group 是否允許、是否 fromMe、文字是否太短
  3. 清理文字，例如去除多餘空白、控制字元
  4. 呼叫 OpenAI Embeddings API
  5. 用 messageId upsert 到 Qdrant
  6. 成功後結束，不回覆 WhatsApp

chat 分支

  1. Webhook 收到 command: chat
  2. 將使用者問題轉 embedding
  3. 用 groupId filter 搜尋 Qdrant top 8
  4. 將結果按相關度與時間整理成 context
  5. 組裝 RAG prompt 給 DeepSeek
  6. 呼叫 whatsapp-bridge 的 /send-message 回覆群組

9. Prompt 設計

System:
你是一個 WhatsApp 群組 AI 助手。
你可以使用群組共同記憶回答問題。
如果記憶不足，請明確說「我目前沒有足夠記憶」。
不要編造沒有出現在記憶裡的內容。
回答請簡潔、自然，使用繁體中文或粵語。

Context:
以下是這個 WhatsApp 群組的相關歷史記憶：

1. [Kelvin, 2026-05-31] 我星期五要去深圳
2. [Alex, 2026-05-30] 我最近想買 Sony 相機
3. [May, 2026-05-29] 我下星期生日

User:
Kelvin 最近有咩安排？

10. 錯誤處理策略

問題 建議處理
OpenAI Embedding API 失敗 n8n retry 2-3 次；仍失敗就記錄 error，不阻塞 WhatsApp bot。
Qdrant 無法連線 record 分支直接失敗並 log；chat 分支回答「暫時無法查詢記憶」。
Qdrant collection 不存在 建立一個 initialization workflow，啟動後先檢查 / 建立 collection。
DeepSeek 回覆失敗 回傳簡短錯誤訊息，不要暴露 API key 或內部 stack trace。

11. 記錄規則與隱私

要記錄

  • 指定 WhatsApp 群組內的文字訊息
  • @ai 問題本身
  • 圖片 caption 文字
  • 至少 5 個字元以上的有效內容

不記錄

  • bot 自己發出的 AI 回覆
  • 空訊息、太短訊息、純 emoji、純哈哈 / ok / yes
  • 貼圖、語音、影片、圖片本身
  • 非指定群組的訊息
建議公告：「這個群組 AI 會記錄群內文字訊息，用於之後回答群內問題。不會記錄語音、影片、圖片本身。如果有人不想被記錄，可以提出。」

12. 實作順序

Phase 1：Qdrant 基礎

  • 在 docker-compose.yml 加 Qdrant service 和 qdrant_storage volume。
  • 執行 docker compose up -d。
  • 確認 http://localhost:6333/dashboard 可以開。
  • 建立 whatsapp_memory collection，size 為 1536。

Phase 2：修改 whatsapp-bridge

  • 普通群訊息 forward 到 n8n，使用 command: record。
  • @ai 使用 command: chat。
  • @ai memory 使用 command: memory。
  • 保留 @aiimg 的 image 流程。
  • 加入 fromMe / bot 自己訊息過濾。

Phase 3：n8n record 分支

  • 清理文字與過濾短訊息。
  • 呼叫 OpenAI Embeddings API。
  • 產生穩定 messageId。
  • upsert 到 Qdrant。

Phase 4：n8n chat 分支

  • 問題轉 embedding。
  • Qdrant search 加 groupId filter。
  • 組裝 RAG prompt。
  • DeepSeek 回答。
  • 透過 /send-message 回覆 WhatsApp。

Phase 5：清理與管理

  • 加入 180 天保留策略。
  • 未來加入 @ai forget me、@ai memory status、@ai export memory。
  • 定期刪除 expiresAt 過期資料。

13. 第一版完成標準

  • 普通群訊息會被記錄，但 bot 不會回覆普通聊天。
  • 只有 @ai 時才查記憶並回答。
  • Qdrant search 必須只搜尋同一個 groupId。
  • bot 自己的回覆不會被記錄。
  • 太短或無意義訊息不會被記錄。
  • 測試案例：A 說「我星期五要去深圳」，B 問「@ai A 星期五去哪？」，AI 能回答 A 星期五要去深圳。

14. 總結

原本的方案方向正確，可以直接實作。這個改良版主要補上實戰會遇到的細節：穩定 message ID、Qdrant 初始化、錯誤處理、Docker 啟動順序、成本估算、資料保留與明確驗收標準。

建議第一版保持簡單：每條有效文字訊息都存，先把完整 RAG 流程跑通，再做 chunking、forget 指令與自動清理。
