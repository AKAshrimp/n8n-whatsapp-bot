n8n WhatsApp Bot 技術學習文檔

這份 HTML 是給你用來「看懂這個專案」的學習地圖：先看架構，再看資料流，再回頭讀程式碼與 n8n workflow。

Node.js whatsapp-web.js Express Docker n8n DeepSeek API

1. 這個 Project 在做什麼？

這個專案是一個 WhatsApp 群組 AI 助手。當指定群組有人發送 @ai 問題，Node.js bot 會攔截訊息，送到 n8n，n8n 再呼叫 DeepSeek API， 最後把 AI 回覆送回「原本發問的同一個 WhatsApp group」。

WhatsApp Group
@ai hi
→
whatsapp-bridge
Node.js
→
n8n Webhook
→
DeepSeek API
→
send-message
回同一群組
核心觀念：WhatsApp bot 不直接呼叫 AI，而是把訊息交給 n8n。 n8n 負責 workflow、AI API、資料轉換與回傳。

嚴謹回覆同一 group： 程式使用 chat.id._serialized 當 group ID，不用猜群組名稱回傳。 所以哪個 group 問，就把該 group 的 ID 傳到 n8n，再由 n8n 回傳給同一個 group。

Per-user Memory 欄位

現在 index.js 會把 group 與 user 資訊一起送到 n8n，讓記憶可以嚴格分開。 

{
  "from": "groupId",
  "groupId": "groupId",
  "groupName": "群組名稱",
  "userId": "使用者ID",
  "memoryKey": "groupId:userId",
  "command": "chat 或 memory",
  "memoryLimit": 10,
  "text": "使用者問題"
}
記憶隔離原則： memory key 必須使用 groupId:userId。 這樣同一個 user 在不同 group 的記憶不會混在一起，同一個 group 的不同 user 也不會混在一起。

加入 per-user memory 後，n8n workflow 建議是五個 node：

POST /whatsapp-trigger
→
Code
Prepare Memory
→
HTTP Request
DeepSeek
→
Code
Save Memory
→
HTTP Request
Send Message

Node 1：Webhook

Method: POST
Path: whatsapp-trigger

Node 2：Prepare Memory Code Node

這個 node 負責根據 memoryKey = groupId:userId 讀取該 user 在該 group 的記憶，並組合 DeepSeek 需要的 messages。

const store = this.getWorkflowStaticData('global');
const body = $json.body;
const key = body.memoryKey;
const history = store[key] || [];

const system = {
  role: 'system',
  content: '你是一個 WhatsApp 群組 AI 助手。請根據該使用者在該群組的上下文回答，使用繁體中文，簡潔自然。'
};

if (body.command === 'memory') {
  return [{
    json: {
      ...body,
      messages: [
        system,
        ...history,
        {
          role: 'user',
          content: '請總結你目前記得關於我的資訊。若沒有記憶，請說目前沒有足夠記憶。'
        }
      ],
      existingMemory: history
    }
  }];
}

return [{
  json: {
    ...body,
    messages: [
      system,
      ...history,
      {
        role: 'user',
        content: body.text
      }
    ],
    existingMemory: history
  }
}];

Node 3：DeepSeek API

Method: POST
URL: https://api.deepseek.com/chat/completions
Headers:
  Authorization: Bearer YOUR_DEEPSEEK_API_KEY
  Content-Type: application/json

Body 不要寫 "messages": "{{ $json.messages }}"，否則 n8n 會把 array 轉成 string，DeepSeek 會報 messages invalid type。

在 Using JSON 欄位中，請用 JSON.stringify() 把 messages 注入成真正 JSON array：

{
  "model": "deepseek-v4-flash",
  "messages": {{ JSON.stringify($json.messages) }},
  "stream": false
}

Node 4：Save Memory Code Node

這個 node 負責把「這次 user 問題 + AI 回答」存回 static data。 目前保留最近 10 筆 message。若 user 發 @ai memory，只總結記憶，不把這次指令寫入 memory。 

const store = this.getWorkflowStaticData('global');

const input = $('Prepare Memory').item.json;
const aiReply = $json.choices[0].message.content;
const limit = input.memoryLimit || 10;

let history = input.existingMemory || [];

if (input.command !== 'memory') {
  history.push({ role: 'user', content: input.text });
  history.push({ role: 'assistant', content: aiReply });
  history = history.slice(-limit);
  store[input.memoryKey] = history;
}

return [{
  json: {
    to: input.groupId,
    message: aiReply
  }
}];

Node 5：回傳 WhatsApp Bridge

Method: POST
URL: http://whatsapp-bridge:3000/send-message
Body Content Type: JSON
Specify Body: Using Fields Below

Body Parameters：

Name: to
Value: {{ $json.to }}

Name: message
Value: {{ $json.message }}
重要： 這個 node 不要手寫 JSON Body。AI 回覆常常包含換行、引號或特殊字元， 手寫 JSON 很容易報 JSON Body is not valid JSON。 使用 Using Fields Below 讓 n8n 自動處理格式最穩。 n8n UI 裡的 Value 是欄位標籤，不要把 Value: 這個字打進輸入框。
Publish 觀念： Listen for test event 是測試模式，使用 /webhook-test/...。 正式 bot 打的是 /webhook/...，所以第一次要 Publish；之後改 workflow 也要 Publish 才會更新正式版本。

6. Per-user Memory 怎麼運作？

AI API 本身不會記住過去對話。所謂 memory，是我們每次呼叫 DeepSeek 前，把該 user 的歷史訊息一起塞進 messages。

記憶 key

memoryKey = groupId + ":" + userId

這樣可以保證：

  • group A + user 1 不等於 group A + user 2
  • group A + user 1 不等於 group B + user 1

記憶格式

[
  { "role": "user", "content": "我叫 Alex" },
  { "role": "assistant", "content": "好的，我會記得你叫 Alex。" }
]

最多保留 10 筆

MEMORY_LIMIT=10 代表每個 groupId:userId 最多保留最近 10 筆 messages。 注意這裡的 10 筆是 message 數，不是 10 輪對話；一輪 user + assistant 會佔 2 筆。

@ai memory 指令

使用者在群組輸入： 

@ai memory

bot 會請 AI 總結「目前記得關於這個 user 的資訊」。這個指令本身不會寫入 memory，避免污染記憶。

8. n8n Nodes 設定總表

這裡記錄目前 workflow 的所有 nodes。完整文字版也已存成： n8n-nodes.md

不要把所有程式碼貼在同一個 node。 Switch command、prepare memory、Deepseek、save memory、 return message、Image API、return image 都是不同 node，各自有自己的欄位。

Workflow 順序

Webhook
→ Switch command
  ├─ chat / memory → prepare memory → Deepseek → save memory → return message
  └─ image → Image API → return image

1. Webhook

HTTP Method: POST
Path: whatsapp-trigger
Authentication: None
Respond: Immediately

2. Switch command

Node type：Switch。根據 Webhook.body.command 分流。

Route 1: command equals image
Route 2: command equals chat
Route 3: command equals memory

連線方式：

image  → Image API → return image
chat   → prepare memory → Deepseek → save memory → return message
memory → prepare memory → Deepseek → save memory → return message

9. 怎麼判斷卡在哪裡？

看到的現象 代表什麼 怎麼查
WhatsApp 沒反應，logs 沒有 [Received] bot 沒收到訊息，可能 group 名稱不在 allow list，或 bot 沒 ready。 docker logs whatsapp-bridge --tail 120
有 [Received]，但沒有 [Forwarded] bridge 打 n8n webhook 失敗。 檢查 N8N_WEBHOOK_URL 和 n8n Publish 狀態。
有 [Forwarded]，但沒有 [Sent] n8n 最後一個 node 沒打到 bridge，或 workflow 沒 publish。 看 n8n Executions 與最後 HTTP Request node。
n8n 顯示 JSON Body invalid HTTP Request node 的 JSON 格式錯誤。 切到 JSON editor，確認逗號、引號、括號。
Chromium profile lock whatsapp-web.js 的 Chromium profile 被鎖住。 用上面的 lock 修復指令。

10. 建議學習路線

  1. 先懂 Docker：看 docker-compose.yml，理解兩個 container 怎麼互相通訊。
  2. 再懂 HTTP：理解 Webhook、POST、JSON body、Header。
  3. 再看 n8n：理解每個 node 的 input/output JSON。
  4. 最後看 JavaScript：從 index.js 的 event listener、async/await、Express route 開始。
  5. 每次只改一件事：改完先看 logs，再測 WhatsApp。
你學這個 project 的重點不是背 JS 語法， 而是理解「訊息從哪裡來、被轉成什麼 JSON、送去哪裡、怎麼回來」。

11. 最小測試流程

  1. 確認 n8n workflow 已 Publish。
  2. 確認 bridge logs 有 WhatsApp Client is ready!。
  3. 在允許的 group 發：@ai hi
  4. 看 logs 是否有：
[Received] Group: ...
[Forwarded] Message sent to n8n webhook.
[Sent] To: ... | Message: ...

如果三行都有，就代表整條鏈路成功。

Memory 測試流程

  1. 在允許的 group 發：@ai 我叫 Alex
  2. 再發：@ai memory
  3. AI 應該能總結它目前記得「你叫 Alex」。
  4. 換另一個 user 或另一個 group 測試，記憶應該分開。
